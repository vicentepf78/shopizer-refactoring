# ADR-007: InitializationDatabaseImpl permanece no monolito

## Status

Aceito

## Data

2026-07-18

## Contexto

`InitializationDatabaseImpl` orquestra seed multi-domínio (merchant, catalog, tax, user, system). Movê-lo para o reference-service arrastaria domínios não relacionados para a Onda 1. STATE AD-004 mantém o bootstrap no monolito.

## Decisão

Deixar a inicialização do banco / seed multi-domínio no monolito. Os serviços extraídos da Onda 1 assumem que as tabelas de reference e tax já estão populadas (pelo seed do monolito ou por ambientes existentes).

## Alternativas consideradas

### Alternativa 1: Mover o seed para reference-service / tax-service

- **Descrição**: Cada serviço faz seed das suas tabelas no startup.
- **Prós**: Serviços mais autônomos em greenfield.
- **Contras**: Divide um orquestrador cross-domain; duplica premissas de merchant/user.
- **Por que rejeitada**: Fora do escopo da Onda 1; alto risco de divergência de seed.

### Alternativa 2: Job de seed como microsserviço separado na Onda 1

- **Descrição**: Novo serviço de bootstrap dono de todos os seeds.
- **Prós**: Modelo operacional de longo prazo mais limpo.
- **Contras**: Serviço extra sem provar Strangler primeiro.
- **Por que rejeitada**: YAGNI para a onda piloto.

## Consequências

### Positivas

- Onda 1 permanece focada em extração REST e Strangler.
- Comportamento de seed existente preservado.

### Negativas

- Ordem de deploy greenfield: MySQL → (seed do monolito) → reference → tax → monolito strangler.
- reference-service não é totalmente autônomo em DB vazio.

### Riscos

- Operadores sobem serviços contra DB vazio — documentar ordem de startup e dependência de seed nas notas de deploy.

## Notas de implementação

- Não mover packages `init/` / `loader/` de reference necessários apenas para bootstrap se fizerem parte da orquestração de InitializationDatabaseImpl — manter o caminho de bootstrap em `sm-core` / monolito conforme o design.
- O compose de integração deve documentar o pré-requisito de seed.
