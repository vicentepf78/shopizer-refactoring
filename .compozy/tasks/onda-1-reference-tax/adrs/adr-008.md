# ADR-008: Replicação de JWT para o tax-service

## Status

Aceito

## Data

2026-07-18

## Contexto

As APIs privadas de tax exigem o mesmo comportamento de autenticação e autorização por store que o monolito. O DB compartilhado (ADR-002) torna viável o lookup de user/store dentro do tax-service. STATE AD-006 escolheu replicação completa da cadeia JWT. O login permanece no monolito.

## Decisão

Replicar a cadeia de autenticação admin JWT do monolito no `tax-service`: bearer filter, validação JWT com `jwt.secret` compartilhado, lookup de usuário admin e resolução de argumento merchant-store / checks authorize-store. O endpoint de login (`POST /api/v1/private/login`) permanece no monolito. Os endpoints públicos P1 do reference-service permanecem sem autenticação.

## Alternativas consideradas

### Alternativa 1: Auth só no monolito; tax-service confia na rede interna

- **Descrição**: Remover JWT do tax-service; aceitar apenas chamadas do monolito.
- **Prós**: Menos duplicação de código de segurança.
- **Contras**: Defesa em profundidade mais fraca; quebra chamadas admin diretas; diverge do critério de aceite “auth idêntica”.
- **Por que rejeitada**: TAX-05 exige comportamento 401/403 idêntico.

### Alternativa 2: Extrair módulo de security compartilhado na Onda 1

- **Descrição**: Novo JAR `shopizer-security` usado pelo monolito e pelo tax-service.
- **Prós**: DRY no longo prazo.
- **Contras**: Refactor maior tocando a config de security do monolito no piloto.
- **Por que rejeitada**: Preferir copiar/adaptar agora; consolidar se Onda 2+ repetir o padrão.

## Consequências

### Positivas

- Admin pode chamar tax-service via Strangler ou diretamente com o mesmo token.
- Escopo por store aplicado na borda do serviço.

### Negativas

- Configuração de security duplicada para manter.
- Distribuição de secret necessária entre processos.

### Riscos

- Drift de config entre settings JWT do monolito e do tax-service — compartilhar os mesmos valores de env/config; adicionar testes de integração de security (401 sem token).

## Notas de implementação

- Componentes a portar/adaptar: adapter de security da API privada, `AuthenticationTokenFilter`, `JWTAdminAuthenticationManager` / `JWTTokenUtil`, services de usuário admin, `MerchantStoreArgumentResolver`, `LanguageArgumentResolver`.
- O adapter Strangler de tax encaminha o header `Authorization` sem alteração.
