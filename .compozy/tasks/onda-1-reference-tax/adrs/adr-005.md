# ADR-005: DTOs de contrato / sem JPA nas respostas REST; Pact

## Status

Aceito

## Data

2026-07-18

## Contexto

`ReferencesApi` historicamente pode expor entidades JPA (`Language`, `Currency`). Os critérios de sucesso exigem JSON apenas com DTOs. Mappers/populators dependem de entidades e não devem viver em um JAR de contracts publicável (L-002 / OQ-04). Drift cross-service deve falhar no CI (STR-02, TAX-08).

## Decisão

1. Introduzir `shopizer-api-contracts` como JAR livre de JPA (apenas Jackson + validation) contendo DTOs da Onda 1 e interfaces de client.
2. Manter mappers/populators dentro de cada serviço (e temporariamente no monolito até a movimentação).
3. Respostas REST migradas NÃO DEVEM serializar entidades JPA nem grafos lazy (`auditSection`, `stores`, etc.).
4. Usar Pact (JVM): testes consumer no monolito e verificação provider em `reference-service` / `tax-service` para todos os endpoints P1.

## Alternativas consideradas

### Alternativa 1: Colocar mappers no JAR de contracts

- **Descrição**: Compartilhar populators/mappers via módulo de contracts.
- **Prós**: Menos duplicação.
- **Contras**: Puxa `sm-core-model` para contracts; derrota a anti-corrupção.
- **Por que rejeitada**: OQ-04 / L-002.

### Alternativa 2: Contracts só OpenAPI, sem Pact

- **Descrição**: OpenAPI spec-first como único mecanismo de contrato.
- **Prós**: Boa documentação; opções de codegen.
- **Contras**: Por si só não verifica expectativas do consumer no CI.
- **Por que rejeitada**: O TLC exige testes de contrato consumer-driven; OpenAPI pode complementar depois, mas Pact é o gate.

### Alternativa 3: Apenas end-to-end (sem testes de contrato)

- **Descrição**: Confiar em smoke tests com docker-compose.
- **Prós**: Menos ferramentas.
- **Contras**: Lento, frágil, sinal ruim para drift de campos de DTO.
- **Por que rejeitada**: Prática de indústria e TLC STR-02 exigem gates estilo Pact para o template piloto.

## Consequências

### Positivas

- Boundary de contrato publicável clara para ondas posteriores.
- CI captura mudanças de campo que quebram compatibilidade cedo.
- Corrige o vazamento de entidade B-002 em languages/currencies.

### Negativas

- Trabalho de migração/re-export de DTOs em `sm-shop-model`.
- Pact broker/ops pode ser baseado em arquivos locais inicialmente.

### Riscos

- Regras incompletas de matching de campos → contracts frágeis ou frouxos — assertar apenas campos que o consumer precisa; proibir propriedades só-JPA.

## Notas de implementação

- Novo DTO: `ReadableCurrency` (id, code, name, symbol, supported).
- Campos wire de `ReadableLanguage`: apenas id, code, sortOrder.
- Interfaces de client usam códigos string (`storeCode`, `langCode`) — não `MerchantStore`/`Language` JPA no wire.
