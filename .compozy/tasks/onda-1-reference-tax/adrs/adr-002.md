# ADR-002: Schema de banco compartilhado na Onda 1

## Status

Aceito

## Data

2026-07-18

## Contexto

As tabelas de reference e tax participam de foreign keys cross-domain (`TAX_RATE` → country/zone, `PRODUCT` → `TAX_CLASS`, `MERCHANT_STORE` → `LANGUAGE`). Dividir schemas na Onda 1 quebraria o monolito e exigiria refactors de tipos de identidade planejados para a Onda 3. STATE AD-003 já decidiu schema compartilhado para a primeira extração.

## Decisão

Os serviços da Onda 1 e o monolito compartilham o mesmo schema relacional e as mesmas tabelas. A extração é **runtime/API primeiro**, não ownership de dados primeiro. As FKs cross-domain permanecem; sem schemas dedicados por serviço nesta onda (STR-03).

## Alternativas consideradas

### Alternativa 1: Schemas dedicados por serviço na Onda 1

- **Descrição**: Mover tabelas de reference/tax para schemas/databases isolados imediatamente.
- **Prós**: Autonomia de serviço mais limpa; ownership mais claro.
- **Contras**: Quebra FKs JPA; exige trabalho de LanguageCode/MerchantStoreId; alto risco.
- **Por que rejeitada**: Bloqueada por B-001 e FKs de catalog/product; adiada para pós–Onda 3.

### Alternativa 2: CDC / dual-write para DBs separados

- **Descrição**: Manter o DB do monolito como autoritativo e replicar para DBs dos serviços.
- **Prós**: Caminho para ownership eventual sem cortar FKs no dia um.
- **Contras**: Complexidade de infra (CDC, reconciliação) excede os objetivos do piloto da Onda 1.
- **Por que rejeitada**: YAGNI para o piloto Strangler; revisitar ao dividir dados.

## Consequências

### Positivas

- Monolito e serviços mantêm FKs funcionando e os dados de seed existentes.
- Caminho mais rápido para serviços deployáveis e contratos HTTP.

### Negativas

- Não é uma extração pura de dados; os serviços não são totalmente autônomos no storage.
- Acoplamento operacional em migrações e backups de DB permanece.

### Riscos

- Writes acidentais cross-service em tabelas “estrangeiras” — mitigar com boundaries de módulo (`sm-reference-core` / `sm-tax-core`) e code review.
- Times podem pular o split de schema posterior — acompanhar explicitamente no roadmap.

## Notas de implementação

- Mesmo padrão de URL/credenciais de datasource do monolito para ambos os serviços.
- Tabelas com ownership-at-runtime: COUNTRY*, ZONE*, LANGUAGE, CURRENCY (reference); TAX_CLASS, TAX_RATE* (tax).
- O guard de FK de product no delete de tax-class usa a tabela compartilhada `PRODUCT` (count antes do delete).
