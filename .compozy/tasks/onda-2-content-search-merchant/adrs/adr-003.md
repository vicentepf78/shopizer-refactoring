# ADR-003: search-service sem JPA/MySQL

## Status

Aceito

## Data

2026-07-18

## Contexto

Search é um domínio de read-model (~14 arquivos-fonte) já respaldado por OpenSearch via `shopizer-search-opensearch-spring-boot-starter`. Acoplar a MySQL através de `Product` recriaria o problema de indexação do monólito dentro do novo serviço e conflitaria com os objetivos de extração com schema compartilhado.

STATE.md AD-012.

## Decisão

**`search-service` não tem JPA nem DataSource MySQL.** As dependências se limitam a `shopizer-api-contracts`, `shopizer-commons`, o starter OpenSearch e Spring Web/Security/Actuator/Validation. Mutações de índice chegam apenas via HTTP `ProductIndexPayload`. Query/autocomplete leem apenas OpenSearch. Na indisponibilidade do OpenSearch, retornar HTTP 503 — nunca fazer fallback para o DB do catálogo.

## Alternativas consideradas

### Alternativa 1: JPA compartilhado para orquestração de reindex dentro do search-service

- **Descrição**: search-service lista produtos do MySQL para bulk reindex.
- **Prós**: Move o bulk reindex inteiramente para fora do monólito.
- **Contras**: Puxa sm-core-model; acopla o deploy a migrações de DB; contradiz o limite de read-model.
- **Por que rejeitada**: O orchestrator de bulk reindex fica no monólito (dono do catálogo) na Onda 2.

### Alternativa 2: Embutir o client OpenSearch só no monólito, não extrair nada

- **Descrição**: Manter search em sm-shop/sm-core.
- **Prós**: Custo zero de extração.
- **Contras**: Perde o objetivo da Onda 2 e o sequenciamento do plano mestre.
- **Por que rejeitada**: Fora do escopo de produto desta onda.

## Consequências

### Positivas

- Único serviço da Onda 2 que é DB-stateless; scale-out de query mais simples.
- Ownership claro: cluster OpenSearch, mappings e APIs de query vivem em um deployable.

### Negativas

- O caminho de reindex admin no search-service retorna 501; o monólito orquestra listing + producer.

### Riscos

- Índice stale quando eventos de catálogo faltam — documentar GAP-SRCH-*; fixes P3 opcionais apenas.

## Notas de implementação

- Porta 8084; mover `MAPPINGS.json` / `SETTINGS_*.json` e bootstrap para search-service.
- Profile Strangler: o monólito não deve conectar ao OpenSearch.
- IDs de requisito: SRCH-01, SRCH-03, AD-012.
