# ADR-005: APIs internas e X-Internal-Token

## Status

Aceito

## Data

2026-07-18

## Contexto

A Onda 2 precisa de chamadas service-to-service que não devem ser expostas como APIs públicas de vitrine: upsert/delete de índice de search, bytes estáticos e internos de logo de content, store snapshot de merchant para o resolver do BFF. JWT público em todo hop interno é desajeitado para producers (listener de índice) e adiciona latência. Confiança pura de rede sozinha é fraca para mutação de índice.

STATE.md AD-013; design TLC.

## Decisão

Expor endpoints **`/internal/v1/**`** nos serviços content, search e merchant. Protegê-los com **política de cluster/rede** (não roteáveis a partir do ingress público). Para as **APIs de índice de search**, exigir adicionalmente o header **`X-Internal-Token`** batendo com `wave2.search-service.internal-token`. Internos de content/merchant podem depender do isolamento de rede na Onda 2 (token opcional/extensão). Clientes RestTemplate do monólito encaminham correlation id e o token de search onde necessário.

## Alternativas consideradas

### Alternativa 1: JWT em todas as rotas internas

- **Descrição**: Reutilizar JWT de admin para o producer de índice e o resolver.
- **Prós**: Um mecanismo de auth.
- **Contras**: O listener de eventos de catálogo não tem JWT de usuário; service accounts adicionam complexidade.
- **Por que rejeitada**: Encaixe ruim para o producer de índice assíncrono.

### Alternativa 2: Somente mTLS

- **Descrição**: Identidade por certificado entre pods.
- **Prós**: Identidade de serviço forte.
- **Contras**: História local/dev mais pesada; não exigido pelo YAGNI da Onda 2.
- **Por que rejeitada**: Adiado; network policy + token bastam para esta onda.

## Consequências

### Positivas

- Superfície pública vs interna clara; API de índice fora do contrato de vitrine.
- Alinha com o padrão de producer strangler.

### Negativas

- Distribuição/rotação de token é preocupação de ops.
- Força de auth inconsistente (token de search vs content só-rede) precisa ser documentada.

### Riscos

- Ingress mal configurado expondo `/internal/**` — mitigar com checklists de deploy e filters fail-closed em search.

## Notas de implementação

- Search: `POST /internal/v1/index`, `/index/bulk` (máx. 50), `DELETE /internal/v1/index/{productId}`.
- Content: `/internal/v1/static/files/**`, `/internal/v1/content/logo`.
- Merchant: `GET /internal/v1/store/{code}` → `MerchantStoreSnapshot`.
- IDs de requisito: SRCH-04, CNT-08, MCH-04, MCH-07, STR-06.
