# ADR-002: Contrato intermediário de índice ProductIndexPayload

## Status

Aceito

## Data

2026-07-18

## Contexto

`SearchServiceImpl` indexa percorrendo o grafo JPA de `Product` (inventory, pricing, descriptions, variants). AOP in-process + `IndexProductEventListener` não sobrevivem a um split de runtime. O `ProductSnapshot` completo é entregável da Onda 3. Esperar a Onda 3 adiaria a extração de search e perderia uma validação precoce do padrão da Onda 1 em um domínio pequeno.

STATE.md AD-009 e TLC OQ-01 escolheram um DTO intermediário.

## Decisão

Introduzir **`ProductIndexPayload`** (e variante bulk) em `shopizer-api-contracts` com `schemaVersion` (default `1`). O monólito (lado do catálogo) constrói payloads via `ProductIndexPayloadBuilder` e faz POST para as APIs internas de índice do `search-service`. O `search-service` desserializa e indexa sem importar `sm-core-model`. `schemaVersion` não suportado → HTTP 422; payload inválido → HTTP 400.

## Alternativas consideradas

### Alternativa 1: Esperar o ProductSnapshot completo (Onda 3)

- **Descrição**: Adiar search-service até existirem snapshots sistêmicos.
- **Prós**: Um contrato canônico de produto.
- **Contras**: Bloqueia a validação de search na Onda 2; atraso de calendário.
- **Por que rejeitada**: O objetivo da Onda 2 é extrair o caminho de leitura agora; o payload intermediário é a ponte explícita.

### Alternativa 2: search-service carrega Product via JPA compartilhado

- **Descrição**: Dar ao search-service acesso MySQL às tabelas de produto.
- **Prós**: Reutiliza o código de indexação existente com menos trabalho de DTO.
- **Contras**: Acopla search ao schema do catálogo; viola AD-012; bloqueia deploy independente.
- **Por que rejeitada**: Search deve ser DB-stateless na Onda 2.

### Alternativa 3: Message bus / ingestão pull via Kafka

- **Descrição**: Emitir eventos de índice para um stream; OpenSearch ou um consumer faz pull.
- **Prós**: Desacoplamento e backpressure (tendência da indústria).
- **Contras**: Nova infra fora do escopo da Onda 2; YAGNI vs producer HTTP já decidido no TLC.
- **Por que rejeitada**: Fora de escopo; producer HTTP combina com o padrão RestTemplate de AD-005.

## Consequências

### Positivas

- Desbloqueia SRCH-01…04 sem a Onda 3.
- Campo de versão permite evolução em direção a ProductSnapshot.

### Negativas

- O builder permanece acoplado ao catálogo no monólito até ondas posteriores.
- Lacunas conhecidas de reindex permanecem documentadas (GAP-SRCH-*).

### Riscos

- Drift de contrato dual vs ProductSnapshot futuro — mitigar com `schemaVersion` e plano de migração da Onda 3.

## Notas de implementação

- Campos espelham o `IndexItem` atual: id, store, language, name, description, link, image, reviews, brand, category, attributes, variants, inventory, addToCart.
- Monólito: DELETE + POST bulk para `/internal/v1/index*`.
- IDs de requisito: SRCH-04, SRCH-06, SRCH-07.
