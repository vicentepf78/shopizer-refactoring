# ADR-009: Thin proxy no monólito para arquivos estáticos legados

## Status

Aceito

## Data

2026-07-18

## Contexto

Rotas MVC legadas `ImagesController` / `FilesController` servem `/static/files/{storeCode}/{imageType}/{imageName}.{ext}` fora de `/api/v1`. Mover esses controllers para content-service mudaria URLs de deploy ou forçaria rewrites de ingress para toda vitrine. TLC OQ-03 escolheu a Opção B.

## Decisão

Manter os controllers estáticos legados no **BFF do monólito**. Introduzir **`StaticContentProxy`** que busca bytes do content-service `GET /internal/v1/static/files/**` quando o strangler da Onda 2 estiver habilitado. Preservar a semântica de URL de `config.cms.contentUrl` / `LocationImageConfig` para os clientes.

## Alternativas consideradas

### Alternativa 1: Mover controllers estáticos para content-service

- **Descrição**: Servir `/static/files/**` diretamente de content.
- **Prós**: Ownership verdadeiro de content na borda HTTP.
- **Contras**: Churn de ingress/DNS/URL de vitrine na Onda 2.
- **Por que rejeitada**: OQ-03 B — minimizar impacto no cliente.

### Alternativa 2: Cutover CDN na Onda 2

- **Descrição**: Apontar URLs estáticas publicamente para object storage.
- **Prós**: Alivia a origin.
- **Contras**: Auth, arquivos privados e variância de método CMS; fora de escopo.
- **Por que rejeitada**: YAGNI para esta onda.

## Consequências

### Positivas

- URLs da vitrine inalteradas.
- Content ainda é dono dos bytes de blob.

### Negativas

- Hop extra (BFF → content) para assets estáticos.

### Riscos

- Regressão de latência — observar p95 vs orçamento de 2× baseline.

## Notas de implementação

- Depende da API estática interna de content (`C-ready`).
- IDs de requisito: CNT-08, OQ-03.
