# ADR-006: Upload de logo blob-first com compensação

## Status

Aceito

## Data

2026-07-18

## Contexto

O logo da loja hoje atravessa `MerchantStore.storeLogo` (filename no DB) e o storage de blob CMS. Bugs de ordenação (DB antes do blob) deixam lojas apontando para arquivos inexistentes. Após a extração, merchant-service é dono da linha de DB e content-service é dono do blob — consistência cross-service sem saga.

STATE.md AD-014; requisito MCH-04.

## Decisão

**Upload:** POST do blob para content-service primeiro → em sucesso UPDATE `storeLogo` → em falha de DB **compensar** com DELETE do blob; retornar erro ao cliente.  
**Delete:** CLEAR `storeLogo` primeiro → depois DELETE do blob; se o delete de content falhar, aceitar blob órfão com log WARN (mesma classe de tolerância a órfãos do delete de página de content).

Usar `ContentServiceClient` contra a API interna de logo de content.

## Alternativas consideradas

### Alternativa 1: DB primeiro depois blob

- **Descrição**: Persistir filename, depois fazer upload.
- **Prós**: Combina com alguns caminhos legados.
- **Contras**: Modo de falha conhecido deixa referências de logo quebradas.
- **Por que rejeitada**: Corrigindo explicitamente o bug de ordenação na Onda 2.

### Alternativa 2: Saga distribuída / outbox

- **Descrição**: Outbox transacional completo entre serviços.
- **Prós**: Ferramental mais forte de consistência eventual.
- **Contras**: Fora dos Não-objetivos da Onda 2 (saga adiada).
- **Por que rejeitada**: YAGNI; compensação basta para logo.

## Consequências

### Positivas

- Clientes nunca veem um logo set com sucesso sem um blob.
- Semântica clara de falha (500 no caminho de compensate).

### Negativas

- O caminho de delete pode orphanar blobs (aceito).
- Exige a API interna de logo de content antes da feature de logo merchant (gate `C-ready`).

### Riscos

- O próprio delete de compensate falha — logar alto; cleanup operacional.
